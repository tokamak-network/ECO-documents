### **Description**

When updating upgradeable smart contracts for new features or bug fixes, keeping the state variables' declaration order unchanged is essential to avoid storage layout issues.

A practical solution is to include unused state variables or explicitly named storage gaps (like `__gap`) in the base contracts. This foresight allows reserved slots for future use, ensuring that any additions to the contract's state won't disrupt the storage pattern of derived contracts or the compatibility with previously deployed versions. After `ERC-7201`, it is also possible to place all storage variables of a contract into one or more structs like `Namespaced Storage Layout`.

The problem of "Lack of Storage Gap Or NameSpaced Storage Layout in Upgradeable Contract" occurs when **these storage gaps are not incorporated into the base contract's logic nor the base contract defines the namespace storage layout**. ### **Recommendation**

To mitigate this issue:

1. To ensure flexibility for future upgrades to the logic contract, it is recommended to reserve an appropriate storage gap in the base contract. This can be achieved by declaring a fixed-size array, typically of `uint256` elements (each occupying a 32-byte slot), within the base contract. Use a clear identifier, such as `__gap` or a name prefixed with `__gap_`, to explicitly indicate its purpose as reserved storage space.
1. Alternatively, it is recommended to organize all storage variables in the contract into one or more structs, following a pattern such as `Namespaced Storage Layout`, to streamline management and maintain compatibility across upgrades.

It is more appropriate to view the proxy used in the service as being used not only for the concept of upgrade, but also for the concept of logic settings for each function.
In addition, this contract is already used in the service. [the deployed addresses on mainnet ](https://github.com/tokamak-network/ton-staking-v2/blob/ton-staking-v2.0/docs/deployed-addresses-mainnet.md#simple-staking-patch-contracts)
Therefore, reconfiguring these proxies (SeigManagerProxy, DepositManagerProxy, DAOCommitteeProxy) is an option that should be avoided as it would impact the entire existing service.

All logic shares the same memory. When we configure the upgrade logic, we configure the storage to exactly match the storage layout of the previous version, and create additional storage.

We will not tolerate any storage mismatch because we must maintain the existing service. Therefore, we will not introduce the concept of a storage gap.

We don't want to discover problems too late because of storage gaps.  If there is even one mismatch in the storage layout of the upgrade logic, it can cause a fatal error, so it should be detected immediately and the logic should not be upgraded.
