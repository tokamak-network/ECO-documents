Claude Code를 이용한 기존 방식은 짝 프로그래밍에 가깝다. 사람과 AI가 번갈아가며 대화하면서 작업한다.

기존 방식의 문제는 다음과 같다:

- Claude Code 같은 도구는 사람이 계속 옆에서 지켜봐야 한다.
- Claude가 작업 중에 멈춰서 질문하거나 확인을 요청한다.
- 복잡한 문제를 끝까지 해결하지 못한다.

원하는 것은 다음과 같다:

- 사람이 없어도 Claude가 스스로 계속 작업을 한다.
- 하나가 아니라 여러 Claude가 동시에 협력을 한다.

2/

**1단계: Claude를 무한 루프에 넣는다.**

```bash
#!/bin/bash

while true; do
    COMMIT=$(git rev-parse --short=6 HEAD)
    LOGFILE="agent_logs/agent_${COMMIT}.log"

    claude --dangerously-skip-permissions \
           -p "$(cat AGENT_PROMPT.md)" \
           --model claude-opus-X-Y &> "$LOGFILE"
done
```

1. while true — 무한 반복
1. 현재 git 커밋 ID를 가져온다.
1. Claude Code를 실행하고, `AGENT_PROMPT.md` 파일의 내용을 프롬프트로 전달한다.
1. 작업이 끝나면 다시 처음으로 돌아가서 반복한다.

`AGENT_PROMPT.md`에 들어간 지시사항은 다음과 같다:

- 문제를 작은 조각으로 나눠라.
- 현재 무엇을 작업 중인지 기록해라.
- 다음에 뭘 해야 할지 스스로 파악해라.
- 완벽해질 때까지 계속해라.

3/

**2단계: 병렬실행 — 여러 Claude가 동시에 일하게 하기**

단일 Claude는 한 번에 하나의 작업만 처리할 수 있다.

프로젝트가 커지면 여러 문제를 동시에 디버깅하는 것이 더 효율적이다. 병렬 실행을 통해 역할 분담(문서 작성, 코드 품질 관리 등)도 가능하다.

구현 방식은 다음과 같다:

```bash
[upstream] ← 중앙 git 저장소 (빈 bare repo)
    ↑
    ├── [Docker 컨테이너 1] Agent 1 → /workspace (로컬 복사본)
    ├── [Docker 컨테이너 2] Agent 2 → /workspace (로컬 복사본)
    ├── [Docker 컨테이너 3] Agent 3 → /workspace (로컬 복사본)
    └── ... (16개까지)
```

각 에이전트는 다음과 같이 작동한다:

1. 중앙 저장소를 자기 컨테이너에 clone한다.
1. 작업을 수행한다.
1. 완료되면 중앙 저장소로 push한다.
1. 다른 에이전트의 변경사항과 merge한다.

**Lock system**

두 에이전트가 같은 문제를 동시에 풀지 않도록 하기 위해 Lock 시스템을 도입한다.

```bash
current_tasks/
├── parse_if_statement.txt      ← Agent 1이 작업 중
├── codegen_function_def.txt    ← Agent 2가 작업 중
└── ...
```

작동 방식은 다음과 같다:

1. Agent가 작업 시작 전에 current_tasks/작업이름.txt 파일 생성
1. git에 push
1. 다른 Agent가 같은 파일을 만들려고 하면 → git 충돌 발생 → 다른 작업 선택
1. 작업 완료 후 lock 파일 삭제

**중앙에서 "너는 이거 해, 너는 저거 해"라고 지시하는 관리자 AI는 없다. 각 Claude가 스스로 다음 작업을 판단한다.*

**교훈**

코드를 작성하는 것보다 Claude가 **스스로 방향을 잡을 수 있는 환경**을 만드는 데 대부분의 노력이 들어갔다.

Claude는 테스트를 통과하려고 열심히 일을 한다. 만약 테스트가 잘못되면 잘못된 방향으로 열심히 하게 된다.

이를 위해서는 다음이 필요하다:

1. 고품질 테스트 스위트를 확보한다.
1. Claude가 실수하는 패턴을 관찰하고 새 테스트를 추가한다.
1. CI 파이프라인을 구축한다. 새 코드가 기존 기능을 깨뜨리면 거부한다.

사람이 아니라 Claude를 위해 테스트 환경을 설계해야 한다.

**Claude의 한계 1 — 컨텍스트 오염:**

`❌ 나쁜 예: 수천 줄의 로그 출력
✓ 좋은 예: 몇 줄만 출력, 나머지는 파일에 저장
✓ 좋은 예: "ERROR: 이유" 형식으로 grep 가능하게
✓ 좋은 예: 요약 통계를 미리 계산해서 제공`

**Claude의 한계 2 — 시간 감각 없음:**

`❌ 문제: Claude는 시간이 얼마나 지났는지 모름
        → 몇 시간이고 테스트만 돌리면서 진전 없이 시간 낭비

✓ 해결책: --fast 옵션 (1% 또는 10% 샘플만 테스트)
✓ 해결책: 진행 상황을 간헐적으로만 출력`

**Claude의 한계 3 — 매번 새로 시작:**

각 에이전트는 빈 컨테이너에서 시작합니다. 이전 맥락이 없습니다.

`✓ 해결책: README와 progress 파일을 자주 업데이트하도록 지시
          → 새 에이전트가 현재 상황을 빠르게 파악`
