`#단순TEE`, `#PhalaDeRoT` 

***“신뢰의 뿌리가 하드웨어 제조사이다.”***

***"키를 파일에 저장하지 않고, 실행 시점에 하드웨어로부터 직접 유도(Derive)하여 사용"한다.***

소프트웨어 보안이 아니라 하드웨어 보안이다. Intel이나 AMD가 CPU를 만들 때 물리적으로 이 밀실을 칩 안에 넣어둔 것이다.

이 안에 Private Key를 보관하는 것이 아니라 Private Key를 생성하는 기계를 넣어둔다. 이를 위해 2개의 재료가 필요하다.

1. 내가 넣어준 암호문 (salt)
eg. “tokamak-dao-agent-v1”
1. 기계 자체에 내장된 비밀 씨앗 (TEE seed)
eg. 기계에서 꺼낼 수 없음
eg. 밀실 밖으로 나갈 수 없음

재료 1 + 재료 2 + [수학 함수] → Private key 생성

**Remote Attestation**

“우리 서버에 밀실이 있으니 안전합니다.”

이 밀실을 만든 것은 Intel(건설사)이다. Intel은 자기가 만든 진짜 밀실에만 인증서를 발급한다. 이 인증서는 Intel의 서명이 들어가 있어서 위조가 불가능하다.

```bnf
┌─────────────┐        ┌─────────────┐        ┌─────────────┐
│   당신      │        │   밀실      │        │   Intel     │
│  (검증자)   │        │   (TEE)     │        │  (제조사)   │
└─────────────┘        └─────────────┘        └─────────────┘
      │                      │                      │
      │  "진짜 TEE 맞아?"    │                      │
      │ ──────────────────→  │                      │
      │                      │                      │
      │                      │  인증서 발급 요청    │
      │                      │ ──────────────────→  │
      │                      │                      │
      │                      │  Intel 서명된 인증서 │
      │                      │ ←──────────────────  │
      │                      │                      │
      │  인증서 반환         │                      │
      │ ←──────────────────  │                      │
      │                      │                      │
      │                                              │
      │  Intel 공개키로 검증                         │
      │  → "진짜 Intel이 만든 TEE에서               │
      │     이 코드가 돌고 있구나" ✅                │
      │                                              │
```

**How secure?**

1/

`공격자가 할 수 있게 되는 것:
━━━━━━━━━━━━━━━━━━━━━━━━━

✅ 서버의 모든 파일 읽기      (cat .env, cat secret.txt)
✅ 서버의 메모리 읽기          (프로그램이 쓰는 RAM 데이터)
✅ 서버에서 프로그램 실행하기   (자기 코드를 돌릴 수 있음)
✅ 네트워크 트래픽 가로채기

❌ CPU 칩 내부의 물리적 회로를 읽기  ← 이건 불가능`

—> 해킹은 **소프트웨어 레벨**의 공격입니다. 아무리 서버를 완전히 장악해도, CPU 칩 안에 물리적으로 새겨진 값을 소프트웨어로 읽어내는 것은 불가능.

2/

**공격자: 파일 읽자 → cat .env
결과:   WALLET_SECRET_SALT=tokamak-dao-agent-v1
공격자: 이걸로 키를 만들어보자!**

**Private Key = f(salt, TEE 시드)
↑         ↑
갖고 있음   없음!**

**공격자: TEE 시드는 어떻게 구하지?
→ 메모리 덤프? → TEE 영역은 암호화됨, 쓰레기값만 보임
→ 파일 탐색?  → 파일로 존재하지 않음, 하드웨어에 새겨져 있음
→ 프로그램?   → TEE 시드를 출력하는 API 자체가 없음**

**공격자: 키 복원 불가 ❌**

1/ AI 에이전트에게 크립토 지갑을 맡긴다. 근데 에이전트는 원격 서버에서 돌아간다. 그런데 이 서버가 해킹되면 키는 털리게 된다.

2/ 가장 단순한 방법은 서버에 키를 파일로 저장한다. 해커가 서버를 장악해서 cat .env를 실행하면 키는 탈취된다.

3/ 그래서 TEE를 사용한다. TEE는 CPU 안에 있는 격리된 공간이다. OS도, 클라우드 관리자도, 해커도 이 안을 들여다볼 수 없다. 하드웨어 레벨에서 격리되기 때문이다.

4/ TEE 안에서 Docker가 돌아간다. 이 Docker 안에 “키 파생 + 서명” 코드가 있다. TEE 메모리는 암호화되어 있어서, 이 Docker 프로세스만 복호화된 데이터를 볼 수 있다. 밖에서는 쓰레기 값만 보인다.

5/ 외부에서 이 Docker에게 요청할 수 있는 건 “이 트랜잭션에 서명해줘.”이고 서명값만 돌아온다. 키를 꺼내달라는 API는 존재하지 않는다.

**6/** 더 나아가서: 키를 저장하지도 않는다. salt + 비밀 씨앗을 조합해 매번 키를 파생한다. 같은 입력 → 같은 키. 저장할 필요 없음.

**7/** 해커가 salt를 훔쳐도? 비밀 씨앗이 없으면 키 복원 불가. 비밀 씨앗은 TEE 안에만 존재. 소프트웨어 해킹으로 꺼낼 수 없다.

**8/** 여기까지가 기본 TEE. 하지만 문제가 있다. 비밀 씨앗이 CPU 하나에 묶여 있음. 그 CPU가 뚫리거나 고장나면? 끝.

**9/** Phala는 이걸 해결한다. 비밀 씨앗을 CPU 1개에 두지 않고, MPC로 여러 노드에 분산한다. 이게 DeRoT(Decentralized Root-of-Trust).

**10/** 공격자가 키를 복원하려면 노드의 2/3 이상을 동시에 뚫어야 한다. 노드 1개 뚫리면? 블록체인에서 차단. 키 로테이션. 피해 최소화.

**11/** 모든 코드 변경은 블록체인에 기록된다. 배포자가 몰래 악성 코드를 넣어서 키를 빼내는 것이 불가능. 온체인에서 누구나 확인 가능.

**12/** 정리. .env 키 저장 → 서버 해킹 시 즉사. TEE 키 파생 → 서버 해킹에도 키 안전. Phala DeRoT → TEE 하드웨어까지 뚫려도 분산으로 방어. 보안은 레이어다. /end
