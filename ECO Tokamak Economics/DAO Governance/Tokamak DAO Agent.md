## 현재 방식이 최선이 아닌 이유

네 가지 근본적 한계가 있습니다.

**RAG는 이해가 아니라 검색입니다.** ABI 함수 시그니처를 나열하는 건 개발자도 이미 아는 정보입니다. "깊은 이해"란 함수 간 경제적 영향, 시뇨리지 분배 흐름 추적, 과거 제안 패턴 분석 같은 것인데 현재 시스템은 이 중 아무것도 못 합니다.

**500단어 청크 5개로는 컨트랙트 간 관계를 표현할 수 없습니다.** 토카막의 핵심인 SeigManager → DepositManager → Layer2Registry → CoinageFactory 연결을 이해하려면 여러 컨트랙트 맥락이 동시에 필요합니다.

**정적 스냅샷은 이미 오래된 데이터입니다.** 블록체인 상태는 매 블록마다 변하므로 실시간 조회가 필요합니다.

**채팅 UI는 미션과 무관합니다.** 필요한 건 에이전트가 스스로 온체인을 읽고 시뮬레이션하는 능력입니다.

## 제안: Tool 기반 에이전트

`Claude (with deep Tokamak context)
├── read-contract-state     → 실시간 온체인 조회
├── simulate-transaction    → Anvil fork 시뮬레이션
├── analyze-proposal        → calldata 디코딩 + 영향 분석
├── query-history           → 과거 제안/투표 이력
├── read-source-code        → 소스코드 직접 참조
└── compare-state           → 업그레이드 전후 비교`

## 전환 전략

- **유지:** 기존 분석 스크립트들(에이전트 도구로 변환), Solidity 소스코드, 컨트랙트 주소 매핑
- **전환:** RAG → tool-use, 정적 마크다운 → 핵심 도메인 지식만 시스템 프롬프트에, 채팅 UI → CLI/자동화
- **신규:** 실시간 온체인 쿼리, 트랜잭션 시뮬레이션, 제안 모니터링, 소스코드 기반 행동 분석 문서
