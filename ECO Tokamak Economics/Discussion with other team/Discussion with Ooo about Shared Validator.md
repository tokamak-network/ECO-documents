# What is a shared validator

- 검증자 경제 강조, 검증자에게 시뇨리지 분배
- 토카막에서 각 롤업마다 검증자 풀이 별도로 있기는 어려운 구조
- 공통 검증자 풀은 TRH에서 런칭한 모든 롤업의 검증자가 되는 것

# How to use it?

### Ooo state channel

A core element of Tokamak Network's tokenomics is a shared validator pool. On-chain validation of numerous on-demand (RaaS) rollups generated by TRH will be delegated to this single shared validator pool. The pool will be managed by the Random Attention Test protocol, currently being researched and developed by ECO, with TON used for rewards and penalties.

We plan to develop two consensus methods simultaneously and offer users a choice when opening their channel:

1. Consensus via shared validator pool (default): This option does not use threshold signatures between channel users.
  1. Each channel participant generates a ZKP for each transaction batch they create and signs it individually.
  1. The individual ZKPs are submitted to Ethereum *without any proof aggregation*.
  1. The shared validator pool verifies the submitted ZKPs off-chain, reaches consensus, and relays the results to the bridge.
  1. The bridge then updates the state based on the validator pool's consensus.
1. Consensus via threshold signature: This is the approach we've planned so far.
  1. Each channel participant generates a ZKP for each transaction batch they create.
  1. At each rollup phase, the leader aggregates the ZKPs into a single ZKP. 
  1. The channel participants sign the merged ZKP's integrity using the threshold signature.
  1. The signed, aggregated ZKP is submitted to Ethereum.
  1. The bridge then updates the state based on on-chain verification of the aggregated ZKP and the threshold signature.

Both approaches have *trade-offs*:

1. Consensus via shared validator pool: 
  - Pros: Simplifies channel opening and reduces operation costs by removing the need for interactive communication (DKG and threshold signatures) and eliminating on-chain ZKP verification.
  - Cons: Participants must trust the shared validator pool.
1. Consensus via threshold signature: 
  - Pros: No need to trust a third party or the channel leader.
  - Cons: Participants must handle complex communication, and channel operation costs may be higher due to on-chain ZKP verification (assuming shared validator pool fees are low).

Tokamak Network 토큰노믹스의 핵심 요소는 공유 검증자 풀입니다. TRH에서 생성된 수많은 온디맨드(RaaS) 롤업의 온체인 검증은 이 단일 공유 검증자 풀에 위임됩니다. 이 풀은 현재 ECO에서 연구 개발 중인 Random Attention Test 프로토콜로 관리되며, 보상과 페널티에는 TON이 사용됩니다.

Ooo → Pool: zero-knowledge proof (ZKP), public input. Verify(ZKP, public input) = true/false.

우리는 두 가지 합의 방식을 동시에 개발하고, 사용자가 채널을 열 때 선택할 수 있도록 할 계획입니다:

  1. 공유 검증자 풀을 통한 합의(기본값): 이 옵션은 채널 사용자 간 임계값 서명을 사용하지 않습니다.
    1. 각 채널 참여자는 자신이 생성한 각 트랜잭션 배치에 대해 ZKP를 생성하고 개별적으로 서명합니다.
    1. 개별 ZKP는 *증명 집계 없이* 이더리움에 제출됩니다.
    1. 공유 검증자 풀은 제출된 ZKP를 오프체인에서 검증하고, 합의에 도달한 후 그 결과를 브리지에 전달합니다.
    1. 브리지는 검증자 풀의 합의를 기반으로 상태를 업데이트합니다.
  1. 임계값 서명을 통한 합의: 이것은 우리가 지금까지 계획해온 접근 방식입니다.
    1. 각 채널 참여자는 자신이 생성한 각 트랜잭션 배치에 대해 ZKP를 생성합니다.
    1. 각 롤업 단계에서 리더는 ZKP들을 단일 ZKP로 집계합니다.
    1. 채널 참여자들은 병합된 ZKP의 무결성을 임계값 서명을 사용하여 서명합니다.
    1. 서명된 집계 ZKP가 이더리움에 제출됩니다.
    1. 브리지는 집계된 ZKP의 온체인 검증과 임계값 서명을 기반으로 상태를 업데이트합니다.

두 접근 방식 모두 *트레이드오프*가 있습니다:

  1. 공유 검증자 풀을 통한 합의:
    - 장점: 상호작용 통신(DKG 및 임계값 서명)의 필요성을 제거하고 온체인 ZKP 검증을 없애 채널 개설을 단순화하고 운영 비용을 절감합니다.
    - 단점: 참여자들은 공유 검증자 풀을 신뢰해야 합니다.
  1. 임계값 서명을 통한 합의:
    - 장점: 제3자나 채널 리더를 신뢰할 필요가 없습니다.
    - 단점: 참여자들은 복잡한 통신을 처리해야 하며, 온체인 ZKP 검증으로 인해 채널 운영 비용이 더 높을 수 있습니다(공유 검증자 풀 수수료가 낮다고 가정할 경우).
