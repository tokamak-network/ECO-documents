## 🔧 RAT (Randomized Attention Test) 프로토콜

### 핵심 아이디어

- **확률적 도전**: 검증자를 무작위로 선택하여 주기적으로 테스트
- **사전 검증**: 사기 발생 전에 검증자의 준비 상태 확인
- **경제적 인센티브**: 실패 시 직접적인 경제적 페널티

### Attention Puzzle 메커니즘

- **Commitment Binding**: 특정 상태 커밋먼트에 고유하게 바인딩
- **Knowledge Proof**: 올바른 L2 상태 정보를 가진 검증자만 해결 가능
- **효율성**: 두 개의 자식 해시 (LV, RV)만 제출하면 됨

### RAT

```
1. RAT 컨트랙은 L1에 배포한다.
2. RAT 컨트랙은은 챌린저를 위한 스테이킹 함수를 지원한다.
    2.1. 챌린저들은 1(?) ETH 스테이킹을 해야 RAT시스템에 참여할 수 있다.
3. RAT은 챌린저를 {아이디, 챌린저 Address, 스테이킹 양} 구조로 정보를 저장한다.
    2.1. 챌린저 아이디로 정보를 가져올수있고, 챌린저 Address로 알수도 있고, 네트웤 주소로도 정보을 알수있는 인터페이스를 지원한다.
    2.2. 챌린저 목록을 알 수 잇다.
4. RAT 컨트랙에는 파라미터를 입력받아, 특정 챌린저에게 Attention Test 를 하는 이벤트를 발생하는 함수가 있다. (onAttentionTrigger)

```

### 프로토콜 흐름

```
1. Proposer가 L2 상태 전환 계산 -> 기존 옵티미즘에서 구현되어 있는 기능
2. L1 스마트 컨트랙트에 상태 커밋먼트 제출 createGame -> 기존 옵티미즘에서 구현되어 있는 기능 수정 
   -> 게시하는 함수를 수정해서, 3번 트랜잭션이 같이 실행되게 한다. :  sigma P == state root
   -> gamefactory 에 RAT컨트랙 주소 저장  
   
3. **Proposer가 πa 확률로 RAT 컨트랙에 Attention Puzzle 트랜잭션 RAT.onAttentionTrigger(L2blocknumber, sigma P) 을 실행한다.**
    3.1. RAT.onAttentionTrigger(sigma P - state root) 함수는 L2 block number 와 sigma P 를 이용하여 어텐션 할 챌린저를 결정하고 OnAttentionTrigger (challengerId, L2 block number) Event를 발생시킨다. 
        -> blockhash 정보로 validator 리스트에서 한명 선정 
           ex) validator index = blockhbash mod |validator set|
           blockhbash 를 작은값으로 가공해서 모듈러연산
        -> 해당 챌린저의 보증금에서 먼저 임시 삭감한다. (나중에 정답제출시 원복) 
        
4. onAttentionTrigger 함수에 따라 지목된 챌린저는 submitAttentionProof(proof) 트랜잭션을 제출하여야 한다.
    4.1. onAttentionTrigger 함수가 발생되고 언제까지 제출되어야 하는가? -> 시간은 휴리스틱하게 정해야할 것 같습니다. 일단 1시간 정도에 해당하는 블럭넘버 차이로 가면 어떨까 합니다.
    
    4.2. 제출한 검증값이 틀렸는지, 맞았는지 체크하는 방법?  두 개의 자식 해시 (LV, RV)만 제출, 구체적인 방법을 찾아봐야함. 어디서 값을 어떻게 찾는가? -> 
    맞았는지는 컨트랙트가 체크합니다. 두개의 해시값을 조합하여 해시했을때 state root 값이 도출되면 맞는 것으로 간주합니다. 
    
    4.3. 제출여부를 누가 체크하는가? -> 다른 챌린저들이 체크하는가? 또는 프로포저가 체크하는가? (?)
	     => ~~컨트랙이 자동으로 체크...~~ 
	     => 제출여부를 따로 체크할 필요없다. 미리 보증금을 삭감했으므로, 증거를 제출하지 않는다면 보증금을 반환받을수 없다. 
        
    4.3. 기간안데 제출되지 않았다면 어떻게 해야 하는가? -> 기간 안에 제출되지 않았다면 보증금에서 일정 금액이 삭감됩니다 
     -> 트리거시 이미 삭감했음. -> 미리 삭감했고, 기간안에 제출이 안되면, 삭감된 보증금은 받을수없다. 
    
    4.4. 제출한 검증값이 맞았다면 어떻게 해야 하는가? => 제출한 검증값이 맞았다면 아무 액션도 일어나지 않을수도 있는데, 다른 챌린저가 검증했다는게 입증은 안된다. 
       => 임시 삭감된 보증금이 돌아옴. 
       -> 검증값을 제출할때, 올바른 루트 증명 함수를 호출함으로썩 삭감된 보증금을 반환받는다. 
       -> 올바른 루트 증명 데이타를 어떻게 추출하는가? 찾아보자
       
    4.5. 제출한 검증값이 틀렷다면 어떤 일이 일어나는가? -> 제출된 검증값이 일치하지 않으면 dispute game 을 진행하게 됩니다.  -> dispute game 게시하는 함수를 호출한다. 틀린것을 제출하고 바로 시작할밖에 없게. 함수안에서 바로 호출되게... 게임 보증금.. 확인해보자.
    
   
    // --------------
    => 제출된 검증값이 올바르지 않으면 기존 챌린저 기능에서 게임을 자동으로 실행하게 될것이다. 챌린저의 보증금 청구로 인한 resolveClaim 함수가 호출되면 보증금이 반환된다. 
        
    =>  resolveClaim 함수를 통해 게임에서 이기고 이 함수를 호출해서 보증금을 분배받는다면 게임에 참여했다는 증명이 된다. (만약 게임에 하고자 하였으나 챌린저가 많아 게임에 참여하지 못하는 리스크를 안고 가도록 한다. ) 
 
```

### 챌린저

```
1. 챌린저들은 RAT 컨트랙트에 1 ETH 스테이킹을 해야 RAT시스템에 참여할 수 있다.
2. RAT 컨트랙을 모니터링하고 있어야 한다.
3. OnAttentionTrigger (challengerId, L2 block number) Event 에서 챌린저아이디가 본인의 아이디라면
    2.1. 이벤트를 보고 질문을 파악한다. => 어떤 값이 이벤트의 파라미터로 들어가야 하는가? 질문이 무엇인가? -> 이벤트 파라미터는 L2 blocknumber, stateroot, 그리고 validator 주소 정도가 될 것 같습니다
    
    2.2. 챌린저는 답을 어떻게 찾아야 하는가? -> L2 의 트랜잭션을 실행을 통한 state 값을 확인해야합니다. 


```
